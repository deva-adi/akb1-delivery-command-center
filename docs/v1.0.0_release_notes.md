# AKB1 Delivery Command Center v1.0.0
### Release notes | 2026-05-11 | First public release

---

## What this is

A portfolio intelligence dashboard for delivery leaders managing multi-programme, multi-vendor, multi-geo engagements. Fourteen tabs. Six roles. A real intelligence layer on every surface that tells you what is happening, why it is happening, and what to do this week.

Built as a successor to AKB1 Command Center v5.8 with every v5.8 regression locked as a gate from day one.

---

## Who it is for

**Portfolio Owner**: the director who needs to see across all ten programmes in under two minutes. Executive tab, AI Governance, full audit access.

**Delivery Director**: the manager who walks each programme. Delivery Health, Risk and RAID, Ops and SLA, flow visibility, scoped audit trail.

**Programme Manager**: the person accountable for one programme. Delivery Health, Backlog Health, Flow and Velocity, own-programme views.

**Finance Lead**: Financials, Governance Operating Model, AI Governance cadence. Cost and margin lens.

**HR Business Partner**: Workforce Intelligence, Capability and Supply Chain, Governance. People and capability lens.

**Read Only**: all read-access tabs. No mutations, no AP flag.

## Architecture decisions locked at v1.0.0

| Decision | Choice | Rationale |
|----------|--------|-----------|
| JWT library | PyJWT 2.12.1 | python-jose CRITICAL CVE; PyJWT actively maintained |
| Auth strategy | HS256 + Redis lockout + CSRF | Stateless, Redis-backed, no session table needed |
| Seed determinism | numpy RandomState(20260424) | Byte-identical across contributors; locked SHA-256 |
| OpenAPI codegen | FastAPI generates, frontend imports | Single source of truth; CI diffs prevent drift |
| Audit write | Synchronous, best-effort on denial paths | Zero dropped writes on mutations; graceful on FK edge cases |
| Rate limiting | Per-IP per-minute bucket in Redis | Mitigates credential stuffing; raises for benchmark runs |
| Frontend routing | Next.js App Router with edge middleware | Server-side JWT verification before any page renders |

## Known limitations at v1.0.0

Six tabs from the v5.8 roadmap are deferred to v1.1: P and L Cockpit, AI Innovation, Commercial Pipeline, Scenario Planner, Multi-Vendor Scorecard, Change Impact. The eight tabs that shipped cover the highest-traffic delivery leadership use cases.

Stub sections exist within shipped tabs wherever seed entities are not yet populated: financials_monthly, decisions, attrition events, client signals, sprint velocity log, DORA metrics, bench state. Every stub is clearly badged in the UI and marked with a TODO comment in code pointing to the missing entity.

Two Next.js DoS HIGH CVEs (GHSA-h25m-26qc-wcjf, GHSA-q4gf-8mx6-v5v3) have no fix in the 14.x branch. Fix requires Next.js 15 (major version). Both are DoS-only with no data exfiltration risk. Mitigated in production by upstream rate limiting and WAF. Target fix: v1.1.

Rate limiter default (120 req/min per IP) is appropriate for typical production traffic but is exhausted by the E2E test suite fan-out pattern and by benchmark runs. E2E tests require RATE_LIMIT_DEFAULT_REQUESTS=10000 at backend start. Benchmarks require 10000000.

---

## Upgrade path from v5.8

v1.0.0 is not an in-place upgrade. It is a clean rewrite on a new stack (Next.js 14 App Router, FastAPI, Postgres). v5.8 used Streamlit. There is no migration path for v5.8 data. The seed generator produces the canonical ten-programme, three-hundred-person dataset for v1.0.0.
